# Habits — Telegram Mini App

Набор мини-приложений в одном Telegram WebApp: трекер привычек, чек-листы,
дневник, менеджер паролей, ссылки, конвертер валют.

- **Прод:** https://telegram.resager.ru/app/habits/
- **Стек:** Go (stdlib `net/http`, pgx) + PostgreSQL · Vue 3 + Vite + TypeScript
- **Актуальный код:** бэкенд `habits/backend`, фронтенд `habits/frontend/app`,
  деплой `habits/deployment/streaks`. Каталоги `habits/bakend` и
  `habits/frontend/webapp` — предыдущее поколение, не используются.

---

## 1. Функционал

### Главная (`/`)
Стартовая страница-знакомство: плитки всех мини-приложений (показываются
только доступные пользователю — см. «Доступ к страницам»). Сверху —
**глобальный поиск** по данным всех своих страниц (кроме паролей — они не
покидают устройство): категории Tracker, чек-листы и шаблоны, записи дневника,
ссылки (имя/URL/теги), метрики, напоминания, серверы; результаты группируются
по страницам, клик ведёт на страницу. Под плитками — блок «🔔 Ближайшие
напоминания». Последняя открытая вкладка запоминается **по умолчанию**
(отдельной опции на главной больше нет).
Навигация — левое выезжающее меню (тоже фильтруется по доступам).

### 📊 Tracker (`/tracker`)
Трекер привычек. Категории (имя + цвет), у каждой — сетка 8 недель × 7 дней
(недели с понедельника, сегодня подчёркнуто). Клик по клетке — отметка
(оптимистично, с откатом при ошибке). Категории сворачиваются шевроном —
состояние хранится на сервере (`user_settings.ui_collapsed`) и переживает
очистку webview. Календарь-модалка по месяцам с
навигацией и отметками. Настройки категории: переименование, цвет, удаление,
галочка **Daily** — помечает категорию как ежедневную привычку: в Reminders
для неё можно создать напоминание, которое приходит, только если день ещё
не отмечен.

**Виды трекеров** (настройки категории): *отметки* или *счётчик*. Счётчик —
в клетках числа, клик +1, долгое нажатие −1, ниже нуля не уходит (ноль =
отсутствие строки в БД). У отметок три **стиля**: квадратики, кружки, эмодзи.
Опция **мультицвет** (для квадратиков/кружков) и **мульти-эмодзи** (для
эмодзи): в разные даты можно ставить разные цвета/эмодзи — рядом с 📅
появляется иконка активного цвета/эмодзи, по клику пикер (палитра/набор +
последние использованные, хранятся в localStorage; активный выбор — тоже,
`tracker_active_v1`). Клик по отметке другого цвета/эмодзи перекрашивает её в
активный, того же — снимает. У одноцветных трекеров цвет отметки в БД NULL —
смена цвета категории перекрашивает и старые отметки. **Вся история** (кнопка
в настройках) — полноэкранный вид всех отметок помесячно с самой старой
(свежие месяцы сверху, итог «отметок/сумма», клетки кликабельны).
**Совместный доступ**: владелец в настройках выдаёт доступ пользователю
приложения (точный id или @логин, получателю — уведомление от бота, подсказки
из общего списка получателей). Участники видят трекер у себя (бейдж 👥 у
названия), ставят отметки и могут создавать Daily-напоминания, но настройки,
шаринг и удаление — только у владельца; участник может покинуть трекер
(настройки → «Покинуть»), владелец — отозвать доступ (✕ у участника).

### ✅ Checker (`/checker`)
Чек-листы: группы с пунктами. Отметка выполнения, добавление пунктов инлайн,
переименование/удаление группы, удаление пункта (подтверждение вторым тапом).
**Долгое нажатие на пункт (v2.22)** открывает окно редактирования (Teleport-
модалка): правка названия, **выбор группы для переноса пункта** (v2.34 —
`group_id` в `PATCH /checker/items/{id}` → `MoveCheckItem`, список всех групп
полным путём «Родитель › Подгруппа», подгруппы идут сразу под своим родителем) + кнопки «Сохранить»/«Удалить»; тап при этом не
переключает галочку. Группы сворачиваются,
состояние на сервере, как у Tracker. Счётчик «сделано/всего» у заголовка группы
показывается **всегда** (v2.33, раньше — только в свёрнутом виде; считает прямые
пункты группы). В настройках группы (⚙️) — чекбокс **«Скрывать выполненное»**
(`hide_done` на `checker_groups`): выполненные пункты не показываются в списке, но
учитываются в счётчике.
**Подгруппы (v2.21, вложенность любой глубины — v2.32)** — из настроек любой
группы/подгруппы (⚙️) можно добавить подгруппу; она рендерится вложенной
карточкой (рекурсивный `CheckGroupCard`, `parent_id`). У подгрупп **нет
собственной подложки** — только левая линия-акцент и отступ, фон
родителя-карточки (в т.ч. «стекло») просвечивает. Удаление родителя каскадно
уносит всё поддерево (Postgres `ON DELETE CASCADE`, любая глубина). **Предел
вложенности — 20 уровней** (`store.MaxCheckerDepth`, группа верхнего уровня =
1-й), единый для ручного создания (`CreateCheckGroup` считает глубину родителя
рекурсивным CTE → `ErrTooDeep` → 409 `too_deep`), импорта и раскрытия в Projects;
в UI кнопка «➕ Подгруппа» скрывается на 20-м уровне с подсказкой.
**Поиск (v2.32)** — глобальное поле поиска на странице фильтрует группы по имени
и пунктам (включая пункты подгрупп любой глубины); при активном поиске карточки
раскрываются, показываются только совпавшие пункты/ветки, а **совпавшие слова
подсвечиваются** (v2.33, `highlightParts` — сегменты `<span class="hit">`, без
v-html). У каждой группы с подгруппами есть свой 🔍 — локальный поиск по её
поддереву (`search.ts`: `groupRelevant`/`visibleItems`).
**Шаблоны** — многоразовые списки («сборы в поездку»): «запустить» разворачивает
шаблон в свежую группу с неотмеченными пунктами; любую группу можно сохранить
как шаблон из её настроек. Шаблон можно удалить прямо из строки (🗑,
подтверждение вторым тапом) или из окна редактирования.
**Шаринг шаблона / группы**: 1) пользователю приложения по точному id или
@логину (получателю приходит уведомление от бота, копия добавляется ему);
2) **ссылка-приглашение** `t.me/<бот>?startapp=chk_<токен>` (шаблон) или
`startapp=chg_<токен>` (живая группа с подгруппами любой глубины, v2.21) — для любого друга в
Telegram: он открывает ссылку, Mini App запускается и автоматически добавляет
копию (повторное открытие не дублирует). Делиться группой можно из её настроек (⚙️).
**Экспорт/импорт группы (v2.23)** — из настроек группы (⚙️) «⬇️ Экспорт»
показывает группу простым текстом (первая строка — имя, пункты «- …», подгруппы
«# …», глубина = число «#»: «##» — подгруппа 2-го уровня и т.д.) и JSON
`{name, items:[{name,done}], subgroups:[…]}` (рекурсивные `subgroups`). Внизу
списка «⬆️ Импорт»: вставленный текст/JSON распознаётся автоматически
(`transfer.ts`), `POST /checker/groups/import` создаёт дерево любой глубины на
сервере. Текст простой — состояние галочек в нём опускается; JSON сохраняет `done`.
Чек-лист-ссылка в Projects раскрывается рекурсивно (`CheckerGroupView`).

### 🗂 Tasks (`/tasks`)
Задачи: разовые дела со сроками, приоритетами и жизненным циклом. Быстрое
добавление одной строкой (поле + выбор категории). **Категории** (в UI;
в API/БД исторически project) — группы задач (цвет + имя), сворачиваются
(`ui_collapsed`, app `tasks`, «Входящие» = id 0); задачи вне категории —
«Входящие». **Вкладки**: Сегодня (срок сегодня +
просроченные, просрочка красным, приоритет важнее), Планы (группы по датам,
«Просрочено» первой), Все (по категориям, ручной порядок стрелками ▲▼),
Выполненные (лог по дням, клик по кружку возвращает в работу).
**Статусы**: стандартные Открыта/Выполнена/Архив; в категории переопределяются
(до 20, у каждого имя + kind: open/done/archived — kind задаёт поведение:
done перечёркивает и уводит в лог, archived скрывает; нужен минимум один open
и один done). В карточке статус — выпадающий список, переключается в любом
порядке. **Карточка задачи**: статус, приоритет (полоска слева:
зелёный/оранжевый/красный), категория, срок (дата + опц. время), 🔔 напомнить
(за N минут), повторение (daily/weekly/monthly/интервал N дней — при
выполнении создаётся следующий экземпляр со сдвинутым сроком), исполнитель
(в общих категориях), подзадачи-чеклист (прогресс «2/5» на строке), описание
в Markdown (рендер как в Articles). **Настройки категории**: цвет, свои статусы,
параметры новых задач по умолчанию (приоритет, напоминание, повтор),
совместный доступ (по id/@логину, бейдж 👥; участники видят все задачи
категории, добавляют и выполняют их, настройки/удаление — у владельца; участник
может покинуть). **Уведомления ботом** (воркер, раз в минуту): «подходит
срок» в момент срока или за N минут (для задач без времени срок = 09:00
пользователя) и один раз «просрочена» (без времени — после конца дня);
получатель — исполнитель или автор. На главной — бейдж на плитке
(сегодня + просрочено, красный при просрочке) и секция в глобальном поиске.

### 📔 Diary (`/diary`)
Дневник: записи с датой-временем, редактирование, удаление, живой поиск
(debounce 300 мс, ILIKE на сервере), фильтр по диапазону дат.
Длинные записи (>400 символов или >6 строк) в списке укорачиваются
(line-clamp на 6 строк) и разворачиваются/сворачиваются кнопкой (v2.17).
Экспорт в .txt/.csv за период — в Settings (генерация на клиенте).

### 🔑 Passwords (`/passwords`) — E2E-шифрование + синхронизация
Менеджер паролей. Шифрование **всегда на устройстве**: PBKDF2-SHA256
(310k итераций) → AES-256-GCM; мастер-пароль не покидает устройство.
**Синхронизация через сервер**: один зашифрованный блоб на пользователя
(`password_vaults`) с **версионированием** (optimistic-блокировка — PUT со
старой версией получает 409, клиент подтягивает серверную и предупреждает,
что изменение с «отставшего» устройства не применено). localStorage — кэш
на случай недоступности сервера: оффлайн-правки помечаются dirty и
досылаются; старые локальные хранилища мигрируют на сервер при первом входе.
На странице показывается **версия и время последнего дампа** на сервер.
- **Папки: вложенные** (дерево), **закрепляемые** 📌 (показываются первыми);
  при удалении содержимое переносится на уровень выше.
- **Передача папки пользователю** по id/@логину: поддерево с паролями
  шифруется на устройстве отправителя случайным ключом, получатель видит
  баннер «Принять/Отклонить» и получает уведомление от бота.
- Записи: **обязательно только название**; логин, пароль, **ссылка**
  (кнопка 🌐 перейти), описание, TOTP. **Поиск** по названию/логину/ссылке/
  описанию. **Порядок** меняется кнопками ⬆️⬇️ в редактировании.
- **Индикатор надёжности пароля**: полоска красный → оранжевый → жёлтый →
  зелёный (оценка по энтропии), работает и для мастер-пароля при создании.
- **Генератор паролей**: длина 8–64, наборы символов, crypto.getRandomValues
  без modulo bias, минимум по одному символу из каждого набора.
- **TOTP (2FA)**: base32-секрет или otpauth:// URL — живой 6-значный код
  (RFC 6238) с отсчётом, тап — копирование.
- **Автоблокировка по бездействию** — опциональная (выкл/1/5/15 мин).
Экспорт/импорт в Settings: файл **опционально шифруется** (AES-256-GCM,
пароль файла может отличаться от мастер-пароля); импорт понимает
зашифрованный и открытый формат, заменяет хранилище по подтверждению.

### 🔗 Links (`/links`)
Менеджер ссылок: папки с произвольной вложенностью, ссылки с тегами,
избранное ⭐, топ-10 📈 по количеству переходов, поиск по названию/URL/тегам,
подсказки тегов, «вставить из буфера», экспорт папки/всего дерева в JSON,
импорт JSON и списком URL (с выбором папки, общими тегами и «умными» тегами по
домену).
**«Поделиться» — попап как в Checker** (у папки и у отдельной ссылки, кнопка
📤): три способа в одном экране —
- *Пользователю приложения* (по id/@логину) — получатель получает **копию**
  (папка → всё поддерево, ссылка → сама ссылка в корень) через общий
  `deliverShare` (`POST /links/folders/{id}/share` kind `links_folder`,
  `POST /links/{id}/share` kind `link`): сразу при auto_accept, иначе во
  «Входящие» на Contacts;
- *Ссылка-приглашение* `t.me/<bot>?startapp=lnf_<token>` (папка) /
  `lnk_<token>` (ссылка) — друг открывает её и получает копию (redeem-эндпоинты
  `/links/folders/redeem`, `/links/redeem`; токен на объекте, `share_token`);
- *Копировать текстом* — прежнее поведение (папка → список ссылок текстом,
  ссылка → её URL).
Отправка по id и ссылка-приглашение работают только с **серверным** хранилищем
(у объекта должен быть серверный id); копирование текстом — всегда.
**Хранилище на выбор** (Settings): локально (localStorage) или на сервере;
кнопка переноса данных между хранилищами. Сам выбор хранится **на сервере**
(`user_settings.links_storage`): Telegram-webview периодически очищает
localStorage, из-за чего настройка «слетала» на локальное хранилище.
Кнопка «вставить из буфера» работает каскадом: Clipboard API → Telegram
`readTextFromClipboard` → поле ручной вставки (долгое нажатие → «Вставить»).
**Проверка битых ссылок** (опция `links.dead_check`, выдаётся админом
персонально): воркер раз в час обходит ссылки (повторная проверка каждой —
раз в 12 ч, HEAD с фолбэком на GET), мёртвые помечаются 💀 в списке;
401/403/429 живыми считаются, смена URL сбрасывает пометку.

### 📈 Metrics (`/metrics`)
Произвольные метрики: категории («Здоровье»), внутри — элементы-графики
(«Вес», «Сон», «SpO2»). Точка добавляется на текущее время по умолчанию,
можно задать любую дату (в т.ч. в прошлое — встанет в середину графика).
Под графиком — список точек (клик — редактирование/удаление).
Категории и элементы сворачиваются (состояние — локально).
**Типы графиков хранятся на бэкенде отдельной сущностью**
(`metrics_chart_types`: line/bars/tubes; новые типы — новой строкой):
- *Линейный*: несколько серий разными цветами, цвет узлов отдельно от линии,
  градиент-заливка под линией.
- *Столбики*: настраиваемые цвета/ширина/зазор; мультикомпонентные точки
  рисуются сегментами (первый компонент — весь столбец, остальные снизу
  вверх — например сон: всего/глубокий/быстрый).
- *Трубки* (MiFit-style): тонкие вертикальные трубки, внутри окрашенный
  сегмент диапазона min–max (например SpO2).
Фон графика: без цвета — прозрачный, можно цвет или градиент.
Конфигурация элемента — JSONB (`config`): компоненты (ключ/название/цвет)
и настройки типа.

### 🔔 Reminders (`/reminders`)
Напоминания, которые приходят **сообщением от бота** в Telegram (тем же ботом,
что открывает Mini App; пользователь должен хотя бы раз запустить бота).
Типы (kind):
- *once* — один раз в указанные дату и время;
- *daily* — каждый день в заданное время;
- *weekly* — по выбранным дням недели (чипы Пн–Вс, битовая маска);
- *monthly* — в число месяца (если в месяце нет 31-го — последний день);
- *yearly* (v2.17) — ежегодно в дату (день + месяц + время) — для праздников
  и дней рождений; 29 февраля в невисокосный год сдвигается на 28-е;
- *interval* — каждые N минут/часов/дней/недель/месяцев (единица выбирается
  в форме, хранится в минутах, месяц = 30 дней; минимум 5 минут, максимум год);
- *tracker* — триггер по привычке из Tracker: приходит в заданное время
  **только если категория ещё не отмечена за сегодня** (день считается в
  таймзоне пользователя); в выборе категорий daily-помеченные показаны первыми.
Каждое напоминание можно выключить/включить тумблером (без удаления),
у включённых видно следующее срабатывание («сегодня 21:00», «завтра 09:00»).
**Свои категории** (v2.17, `reminder_categories` + `reminders.group_id`):
напоминания раскладываются по сворачивающимся группам («Без категории» —
отдельная группа; сворачивание хранится на сервере, app='reminders');
удаление категории не трогает напоминания (FK SET NULL). Кнопка вверху
переключает **компактный режим** списка (localStorage).
**Экспорт/импорт/шаринг категории (v2.23)** — из настроек категории (⚙️):
«⬇️ Экспорт» (читаемая сводка + JSON `{name, reminders:[…]}`), «📤 Поделиться»
(пользователю приложения или ссылка-приглашение `startapp=rem_<токен>`). Внизу
списка «⬆️ Импорт категории (JSON)» → `POST /reminder-categories/import`.
Копирование/импорт переносит напоминания с пересчётом `next_fire_at`, **кроме
kind=tracker** (завязаны на привычки Tracker получателя). `share_token` —
столбец `reminder_categories`.
Время задаётся в локальной таймзоне пользователя (`tz_offset_minutes`
сохраняется с напоминанием, сервер пересчитывает в UTC).
Фоновый воркер (`internal/reminders`) раз в 30 с выбирает наступившие
(`next_fire_at <= now`), шлёт `sendMessage` через Bot API и планирует
следующее срабатывание; once после отправки выключается. Ошибка отправки
(бот не запущен/заблокирован) не блокирует перенос.

### 📄 Articles (`/articles`)
Текстовые статьи с **Markdown-разметкой** (GFM, как README на GitHub:
заголовки, таблицы, код, цитаты, списки задач; рендер — marked + DOMPurify).
**Вложенные категории** как в Links. Клик по статье открывает её **на весь
экран** (заголовок сверху, крестик закрыть); ссылки из статьи открываются
через Telegram. У каждой статьи меню ⋮: *изменить* (редактор с
markdown-текстом и выбором категории), *скачать* (.md на устройство),
*поделиться*. Шаринг тремя способами: пользователю приложения по id/@логину
(копия + уведомление от бота), ссылка-приглашение
`t.me/<бот>?startapp=art_<токен>` (статья добавится получателю при открытии)
и **публичная ссылка для скачивания** `…/dl/articles/<токен>` — отдаёт .md
без авторизации, работает у кого угодно. Лимит статьи — 1 МБ.
**Поиск** в двух режимах: «названия и категории» (мгновенно, по дереву) и
«по содержимому» (сервер, со сниппетом вокруг совпадения; ищет и в доступных
категориях). **Шаринг целой категории** — доступом, без дублирования
контента (`articles_folder_shares`): получатель видит поддерево владельца
вживую (обновления автоматически), read-only (в меню только «скачать»),
секция «📥 Доступные мне»; владелец управляет списком доступа в настройках
категории, получатель может убрать категорию у себя. **Редактор — на весь
экран** (заголовок, категория, растянутое поле текста), в шапке кнопка «?»
со шпаргалкой по Markdown-разметке.

**Articles v2 (v2.17):**
- меню ⋮ дополнено: *копировать* (вся статья в буфер + тост «Скопировано»),
  *информация* (размер Б/КБ/МБ, число символов, даты создания/изменения),
  *история*;
- **история изменений как git**: при каждом сохранении с изменённым текстом
  старая версия уходит в `article_revisions` (хранятся последние 30);
  экран истории показывает список ревизий (дата + размер), клик — diff
  «что изменилось с той версии до текущей» (построчный LCS с контекстом,
  `src/apps/articles/diff.ts`, +зелёным/−красным);
- **публичная ссылка на чтение**: `…/read/<токен>` (read_token, отдельный от
  скачивания) — SPA-роут `/read/:token` рендерит статью в браузере без
  авторизации (JSON с `GET /api/v1/articles/public/{token}` вне auth);
- **позиция чтения**: читалка запоминает долю прокрутки на сервере
  (`article_read_positions`, PUT /articles/{id}/read-pos, debounce 800 мс
  + сохранение при закрытии) и восстанавливает при открытии;
- **двойной тап** по тексту в читалке открывает редактор, прокрученный к
  этому месту (поиск начала тапнутого блока в сыром Markdown, курсор туда);
- **картинки**: кнопка 🖼 в редакторе загружает файл (jpeg/png/webp/gif,
  до 5 МБ) на `POST /articles/images` → `data/articles/<random>.<ext>`,
  раздаётся публично `/uploads/articles/`; в текст вставляется
  `![картинка](/app/habits/uploads/articles/…)`, рендерится `<img>`.

### 🖥 Servers (`/servers`) — по умолчанию скрыта
Мониторинг серверов; видимость по умолчанию «персональный доступ» (выдаёт
админ). «Добавить сервер» — имя + адрес агента (+ опциональный токен).
На каждый сервер ставится **habits-agent** (`habits/agent`, один статический
бинарник, только stdlib и /proc): отдаёт `GET /metrics` с ОС, внешним IP,
uptime, CPU (%, ядра, load1), RAM и списком дисков (всего/свободно);
защита — `Authorization: Bearer <токен>`. Бэкенд-поллер опрашивает адреса
раз в минуту, хранит снапшот и историю CPU/RAM за 48 ч. Карточка сервера:
статус-точка, метаданные, **графики CPU и RAM за 24 ч**, диски с
прогресс-барами (красный от 85 %), ошибки опроса, ручное обновление 🔄.
Установка агента: `deployment/streaks/deploy-agent.sh` (systemd, порт 9101
открыт только докер-подсетям).

**Компактный режим (v2.21)** — переключатель справа над списком (клиентская
настройка в `localStorage`, ключ `servers_compact`): скрывает строку
метаданных, объединяет CPU и RAM в один совмещённый график без подписей
(`ComboChart`) и показывает диски компактно (тонкие бары). Подробный режим —
две отдельные спарклайн-карточки, как раньше.

**Пороговые уведомления по серверу (v2.21, диски по-раздельно v2.22)** — в
настройках сервера (⚙️) раздел «🔔 Уведомлять о лимитах»: **диски по каждой
точке монтирования отдельно** (чекбокс «мониторить» + свой порог свободного
места в ГБ на каждый раздел — чтобы не алертить `/boot/efi` и подобные); RAM
выше N % дольше M минут; CPU выше N % дольше M минут. Пороги хранятся в
столбцах `servers` (`alerts_enabled`, `disk_rules` JSONB `[{mount,min_free_mb}]`,
`disk_min_free_mb` — порог по умолчанию для UI, `ram_pct`/`ram_minutes`,
`cpu_pct`/`cpu_minutes`). Поллер раз в минуту оценивает условия: каждый
мониторимый диск — по последнему снапшоту, RAM/CPU — по устойчивому превышению
на всём окне (`server_samples`, покрытие ≥ 70 % окна). Одно сообщение на
событие: по дискам — множество точек в алерте (`disk_alerted_mounts` JSONB,
сообщение на каждую точку), по RAM/CPU — флаги `ram_alerted`/`cpu_alerted`; при
возврате в норму — «🟢 … в норме». Offline-серверы не оцениваются (устаревшие
данные). Изменение порогов сбрасывает состояние алертов.

**Домашние машины (push-режим, v2.16)** — для машин без внешнего IP.
При добавлении выбирается «🏠 Домашняя машина»: сервер генерирует
`push_token` и показывает команду установки. Агент с заданным
`AGENT_PUSH_URL` сам шлёт тот же JSON-отчёт раз в `AGENT_PUSH_INTERVAL`
(60 с) на `POST /api/v1/agent/push` (вне tma-авторизации, Bearer =
push-токен); входящие порты не нужны, HTTP-сервер агента поднимается
только при явном `AGENT_ADDR`. Поллер такие машины не опрашивает.
Карточка — та же (бейдж 🏠, без кнопки 🔄), online = отчёт моложе 3 минут.
Установка: однострочник из публичного репо агента
(`curl -fsSL …habits-agent/main/install-home.sh | sudo bash -s -- <токен>`,
бинарник берётся из GitHub-релиза) или `deployment/streaks/install-home-agent.sh`
локально из монорепо (сборка из исходников). Публичный репозиторий агента —
github.com/resagera/habits-agent (main.go синхронизируется с `habits/agent`,
релизы с бинарниками amd64/arm64).

**Offline/online-уведомления ботом (v2.16)** — для всех серверов (pull и
push): если `last_ok_at` старше 3 минут, владельцу приходит одно сообщение
«🔴 не выходит на связь» (флаг `offline_notified` защищает от повторов),
при первом удачном отчёте/опросе — «🟢 снова на связи». Серверы, ни разу
не выходившие на связь, не алертятся.

### 📁 My Files (`/files`) — по умолчанию скрыта, v2.18
Доступ к папкам на домашних машинах из телефона; видимость «персональный
доступ» (как Servers). «Добавить машину» — имя, сервер генерирует токен и
показывает команду установки. На машину ставится **habits-files-agent**
(`habits/files-agent`, отдельный бинарник, gorilla/websocket): в его конфиге
`FILES_AGENT_ROOTS` перечисляются папки в виде `путь:ro|rw` через `;` — режим
`ro` (только чтение) или `rw` (чтение и запись). Любой путь запроса
проверяется на принадлежность одной из папок; выход за их пределы (в т.ч.
через `..` и симлинки) запрещён, запись в `ro`-папку отклоняется.

**Транспорт.** У домашних машин нет внешнего IP, поэтому агент держит
**исходящий WebSocket** на `GET /api/v1/files/agent` (вне tma-авторизации,
Bearer = токен машины); первым кадром шлёт `hello` со списком папок (они
кэшируются в БД для показа). Бэкенд — релей (`internal/files`): пришедший от
UI HTTP-запрос спаривается с этим соединением по id машины. Протокол
запрос/ответ мультиплексируется по `id`; данные чтения приходят **бинарными
кадрами** (8-байтовый префикс = id запроса), управляющие ответы — JSON.
Одно активное соединение на машину (новое вытесняет старое), ping/pong
каждые 30 с, реконнект агента с экспоненциальным бэкоффом.

**UI.** Карточка машины со статус-точкой (online = отчёт агента активен),
сворачивается. Внутри — браузер папок с хлебными крошками; клик по папке
углубляется, по файлу — открывает просмотр. Просмотр: тексты (`<pre>`, до
2 МБ), картинки (`<img>`), музыка (`<audio>`), видео (`<video>`), PDF
(`<iframe>`) — всё **потоком через бэкенд** с поддержкой HTTP Range (перемотка
видео/аудио, дозагрузка). В `rw`-папках: загрузка файла (сырым телом,
бэкенд пишет чанками по 512 КБ), создание папки, переименование, удаление;
у каждой записи меню ⋮ (скачать/переименовать/удалить).

**Стриминг-пропуски.** `<video>/<audio>/<img>` в вебвью не шлют заголовок
`Authorization`, поэтому UI сначала запрашивает `POST /files/machines/{id}/ticket`
(bound к машине и пути), получает одноразовый токен и грузит файл по
`GET /api/v1/files/stream/{ticket}` (вне tma-авторизации, авторизация — сам
пропуск, TTL 6 ч). Отдача — с `Accept-Ranges`, `Content-Range`,
`Content-Disposition` (для скачивания); бэкенд читает у агента чанками и
пишет в ответ с backpressure.

Установка: однострочник из репо агента
(`curl -fsSL …habits-agent/main/install-files.sh | sudo bash -s -- <токен> "<папки>"`).

### ⌨️ Terminal (`/terminal`) — по умолчанию скрыта, v2.20
Веб-консоль (shell) к домашней машине из любой точки; видимость «персональный
доступ». «Добавить машину» — имя, сервер генерирует токен и команду установки.
На машину ставится **habits-term-agent** (`habits/term-agent`, gorilla/websocket
+ creack/pty): держит **исходящий WebSocket** на `GET /api/v1/terminal/agent`
(вне tma-авторизации, Bearer = токен машины) и по запросу открывает PTY —
`bash -l` от пользователя агента (переменная `TERM=xterm-256color`, стартовый
каталог настраивается).

**Транспорт.** Бэкенд-релей (`internal/terminal`) мультиплексирует несколько
сессий по одному WS агента: у каждой свой id, кадры данных бинарные с
8-байтовым префиксом-id, управляющие (`open`/`resize`/`close`/`exit`) — JSON.
Браузер открывает свой WebSocket `GET /api/v1/terminal/stream/{ticket}` (вне
tma-авторизации; ticket одноразовый, TTL 60 с, выдаётся `POST
/terminal/machines/{id}/session` владельцу онлайн-машины). Кадры браузера:
бинарные — stdin, текстовые — `{"t":"resize",cols,rows}`; к браузеру бинарные —
stdout PTY. Хаб на медленном клиенте притормаживает агента (блокирующая
пересылка), при разрыве клиента — шлёт агенту `close`, тот убивает PTY.

**UI.** Терминал — **xterm.js** (ленивый чанк) на весь экран: реальный bash с
цветами, автодополнением, редакторами (vim/nano), запуском `claude` и т.п.
Внизу — панель спецклавиш для мобильной клавиатуры (Esc, Tab, стрелки, ^C/^D/
^Z/^L, `|~/`, и «липкий» Ctrl: следующая буква → Ctrl-символ). Ресайз по
FitAddon шлёт `resize`; при обрыве — кнопка переподключения (новый ticket).

**⚠️ Безопасность.** Полный доступ к shell под пользователем агента. Гарантии:
страница персональная (по умолчанию только админ), токен машины = секрет,
ticket одноразовый. Ставить только на свои машины.

Установка: `curl -fsSL …habits-agent/main/install-term.sh | sudo bash -s -- <токен>`.

### 👥 Contacts (`/contacts`) — v2.24
Контакты и входящие шаринги. Контакт добавляется вручную (id или @логин) или
в один тап из подсказок «вы делились с» (`share_recipients`). У контакта:
**примечание** (textarea, до 2000 символов), **галерея фото** (v2.25: таблица
`contact_photos`, до 20 на контакт; jpeg/png/webp до 5 МБ; файлы под
случайными именами в `/uploads/contacts/`; удаление фото/контакта подчищает
файлы; на фронте — миниатюры + лайтбокс с удалением, мультизагрузка) и
галочка **«Принимать расшаренные данные сразу»** (`auto_accept`).

**Внешние контакты (v2.25)** — можно добавить человека, которого ещё нет в
боте: `contacts.contact_id` становится NULL, известное хранится в `tg_id` /
`ext_username` / `ext_name`. По числовому id бэкенд пробует узнать имя/логин
через Bot API `getChat` (работает, если бот видел пользователя); @логины
частных аккаунтов Bot API не резолвит — контакт отображается как «Новый
контакт» с бейджем «не в боте». Галочка auto_accept для них выключена
(отправлять данные всё равно может только зарегистрированный). **Привязка**:
`TouchUser` (upsert при входе) дополняет `contact_id` у всех контактов, где
совпал `tg_id` или (без регистра) `ext_username`, — карточка сама заполняется
именем/логином из `users`. Дедупликация внешних — по tg_id/логину.

**Входящие** (`incoming_shares`): все send-обработчики приложения (шаблон и
группа Checker, категория Reminders, статья, папка и ссылка Links, доступ к
трекеру, доступ к категории задач) доставляют через общий `deliverShare`: если у получателя
отправитель в контактах с галочкой — применяется сразу (как раньше), иначе
запись попадает во «Входящие» на странице Contacts (бот шлёт «хочет
поделиться…»), где её можно **принять** (копия/доступ создаются в этот момент;
если источник уже удалён отправителем — запись снимается с сообщением) или
**отклонить**. Ссылки-приглашения (`chk_/chg_/rem_/art_/lnf_/lnk_`) не проходят через
входящие — их получатель открывает сам. Передача папок Passwords сохраняет
свой отдельный E2E-флоу «Принять/Отклонить» (сервер не может применить её
автоматически — расшифровка только на устройстве).

### 📦 Projects (`/projects`) — v2.26
Проекты-сборники: проект состоит из **блоков** — текст (обычный или rich
text/Markdown), картинки (одна — во всю ширину, несколько — мини-галерея с
полноэкранным лайтбоксом: pinch-zoom двумя пальцами, дабл-тап 2x, листание),
файлы (до 20 МБ, любой тип), геолокация (широта/долгота/подпись + кнопка
«моё местоположение», ссылка на карту) и **ref-блоки** — живые ссылки на
сущности других страниц: чек-лист Checker (пункты с галочками и подгруппами,
прогресс), статья Articles (спойлером, рендер Markdown), задача и категория
задач Tasks (статус, приоритет, срок, чек-лист). Ref-блоки **резолвятся при
каждой загрузке** проекта (данные всегда свежие; при открытом проекте —
автообновление раз в 60 с); при добавлении можно выбрать существующий элемент
или создать новый (`create_name`) — он появится и на своей странице. Блоки
вставляются на любую позицию («＋ Блок ниже», позиции сдвигаются), текстовый
блок **разбивается по курсору** на два, у блока настраиваемый фон (color),
спойлер-состояние `collapsed` хранится на сервере. Проекты **сворачиваются**
(блоки не грузятся, пока проект свёрнут — список отдаёт только метаданные),
группируются в **категории** (сворачиваемые; проекты вне категорий — общий
список). **Параметры проекта**: название, описание, иконка-эмодзи, цвет,
обложка (upload), тип (текст + подсказки из предустановленных
рабочий/личный/разработка/обучение/мероприятие/покупки и уже упомянутых),
статус (draft/planned/active/paused/done/cancelled/archived), теги, даты
начала/планового завершения, часовой пояс, владелец и даты создания/изменения
(отображаются). **Совместный доступ** — через `deliverShare` (kind `project`,
контакт с галочкой — сразу, иначе «Входящие» Contacts); участники добавляют и
редактируют блоки, параметры и удаление — у владельца, участник может
покинуть. **История изменений** (`project_history`): кто и что добавил,
изменил или удалил (создание, параметры, блоки, участники). Если проект
изменил другой участник после вашего последнего просмотра — в списке красная
★ (`project_views`, отметка при открытии). Файлы — в
`DATA_DIR/projects` (`/uploads/projects/`), при удалении блока/проекта
чистятся с диска (вместе с учётом в `project_uploads`).

**Лимиты и типы пользователей (v2.27)**: у каждого пользователя есть тип —
`regular` (по умолчанию), `vip`, `payed1`, `payed2`; тип назначает админ в
карточке пользователя (Settings → Пользователи). На каждый тип — свои лимиты
(`user_type_limits`, редактируются админом в секции «📐 Лимиты Projects по
типам»): **блоков в одном проекте** (по умолчанию 100, считается по владельцу
проекта), **картинок и файлов суммарно на пользователя** (учёт —
`project_uploads`, запись снимается при удалении блока/проекта), **размер
одной картинки и одного файла в МБ**. Превышение — 400 с человекочитаемым
сообщением (показывается тостом). Кнопка «＋ Блок» есть и сверху списка
блоков (вставка в самое начало), и снизу.

### 🍽 Food (`/food`) — v2.28

Дневник питания с подсчётом калорий и БЖУ. Четыре вкладки (активная — в
`?tab=`, восстанавливается после обновления): **Дневник** (по умолчанию),
**Общие**, **Рецепты**, **Статистика**.

**Дневник**: выбор дня (← → , «Сегодня», календарь), сводка дня с
прогресс-барами калорий и БЖУ (цель/остаток; превышение не обрезается —
красный бар + текст «превышение N»), приёмы пищи группами
завтрак/обед/ужин/перекусы/без категории с итогами группы. Запись: название,
время, описание, фото, состав из продуктов; бейджи 📋/📖 — создана из
шаблона/рецепта. Действия (⋮): редактировать, дублировать, перенести на
дату, сохранить как шаблон, создать рецепт, удалить (с подтверждением).
Запись без состава — ручные КБЖУ.

**Продукты**: личный каталог (название, альт. название, производитель,
категория, фото, КБЖУ на 100 г или 100 мл, вес 1 шт и 1 порции). Поиск в
форме добавления (debounce, недавние/частые — первыми: `used_count`,
`last_used_at`), создание нового продукта прямо из поиска. DELETE
использованного в истории продукта архивирует его (физически не удаляется),
неиспользованного — удаляет. Управление каталогом — на вкладке Рецепты.

**Снимки (архитектурное требование)**: элемент записи/шаблона/рецепта хранит
СНИМОК данных продукта (название, количество, единица, вес, КБЖУ на 100 г и
итоги) — изменение глобального продукта, шаблона или рецепта не меняет
старые записи дневника. Итоги пересчитываются на сервере из снимков
(`grams/100 × per100`), округление — только при отображении.

**Единицы**: г, мл, шт, порции. Для шт/порций вес берётся из продукта
(`piece_grams`/`portion_grams`); если не задан — фактический вес вводится
вручную.

**Профиль и цели (⚙️, onboarding при первом заходе без профиля)**: пол, дата
рождения, рост, вес, целевой вес, % жира, активность
(min/low/medium/high/max — коэффициенты 1.2…1.9 в конфиге
`store.FoodActivityCoef`), цель (похудение/поддержание/набор) с
дефицитом/профицитом. Расчёт: BMR по Mifflin–St Jeor → TDEE → коррекция;
белок = база (текущий/целевой вес, безжировая масса по % жира или ручной
вес) × коэффициент г/кг; жиры 30% калорий, углеводы — остаток. Расчёт
показывается с объяснением (details), можно переопределить вручную;
дисклеймер «не заменяет медицинские рекомендации». Цели хранятся
**периодами** (`food_goals`): новая цель закрывает прошлый период днём
раньше — прошлые даты и статистика считаются по цели своего периода.

**Шаблоны** — приём пищи без даты (создаются из записи или вручную),
добавляются в дневник в один тап (снимок текущего состава, фото
переиспользуется — файл один). **Рецепты**: ингредиенты, шаги, итоговый вес
и порции → КБЖУ на 100 г и на порцию; в дневник — по съеденному весу (нужен
итоговый вес) или по порциям.

**Общие**: доступ к дневнику для другого пользователя через deliverShare
(kind `food`, только чтение) с флагами видимости: цели, фото, описания, вес
— проверяются на бэкенде (`food_shares`). Разделы «Доступны мне» (просмотр
чужого дневника по дням) и «Мой дневник доступен» (флаги, отзыв).

**Статистика**: 7/14/30 дней или произвольный период; карточки (средние
КБЖУ, заполненные дни, % выполнения целей, дни сверх/в рамках) и графики
(SVG): калории с линией цели и средним за 7 дней (превышение — красным),
белки с целью, жиры, углеводы, число приёмов, распределение калорий по
приёмам (стек). Слой для будущей интеграции с Metrics:
`GET /food/metrics/{key}` с ключами `food.calories.daily`,
`food.protein.daily`, `food.fat.daily`, `food.carbs.daily`,
`food.meals.count`, `food.goal.calories`, `food.goal.protein`,
`food.goal.calories_completion`, `food.goal.protein_completion`.

**Фото** — только изображения (проверка MIME) до 10 МБ, случайные имена, в
`DATA_DIR/food` (`/uploads/food/`); один файл может использоваться шаблоном
и несколькими записями — физически удаляется только когда ссылок не осталось
(`FoodPhotoRefCount`).

### 🤖 Автоматизация (`/automation`) — v2.29

Автоматизированные действия по расписанию или вручную. Страница
`visibility=personal` (по умолчанию только админ, доступ выдаётся в админке).
Первый и пока единственный тип — **заказ воды на jur.am** (`kind=jur_water`).

**Сетевой выход через домашний агент.** jur.am за Cloudflare, который отдаёт
датацентр-IP прод-сервера 403-челлендж (даже на главной, с любым UA). Поэтому
HTTP-запросы к jur.am **туннелируются через агента на домашней машине**
(резидентный IP проходит). Вся логика сценария остаётся на проде — агент
`habits/automation-agent` (public repo `habits-agent`, subdir `automation-agent`)
держит исходящий WS `GET /api/v1/automation/agent` (Bearer-токен из
`automation_agents`, вне tma-auth) и выполняет одиночные HTTP-запросы к белому
списку (`jur.am`), возвращая ответ как есть (без своих cookie и без следования
редиректам — этим управляет сервер). Прод-хаб `internal/egress` даёт
`http.RoundTripper`, который сериализует запрос в WS-кадр; cookie-jar и
редиректы `jurwater` работают прозрачно поверх туннеля. Без активного агента
запуск невозможен (`Execute` вернёт «домашний агент не в сети»; воркер
откладывает срабатывание на час и шлёт предупреждение). Токен и команда
установки — в карточке (`GET /automation/agent-info`), статус online —
`hub.Online(userID)`.

**Исполнитель** (`internal/jurwater`, `Transport` = релей-транспорт агента)
выполняет сценарий обычными HTTP-запросами с cookie-сессией, **без
headless-браузера** (jur.am — Magento 2 + кастомный модуль `Jur_Checkout`).
Шаги: логин (`/customer/account/loginPost`,
form_key берётся со свежей страницы корзины — Magento меняет его после входа) →
очистка корзины → добавление N бутылей «Макур Джур 19л» (product 465, форма
`/checkout/cart/add`) → чтение долга возвратной тары из CRM
(`/crmintegration/index/bottlesdebt` → `makurJur19`) → адрес и время доставки
(`/rest/ru/V1/carts/mine/shipping-information` с
`extension_attributes.mp_delivery_date/mp_delivery_time`, дата — ближайшая
доступная из `custom_date_off_checkout`, слот — первый из
`custom_time_checkout`) → возвратная тара (кастомный
`/checkoutjur/quote/index`, поле `makurmec`) → оформление наличными
(`/rest/ru/V1/carts/mine/payment-information`, метод `checkmo`). В **dry-run**
последний шаг пропускается (доходит до оформления, заказ не создаётся). Каждый
шаг пишется в лог `[{name, ok, detail}]`.
Шаг оформления считается успешным, **только если сайт вернул числовой номер
заказа** (Magento отдаёт голый id); при 200 с не-числовым телом (null/ошибка)
шаг падает с сырым ответом — иначе «нет ошибок, но и заказа нет» (безопасно для
повтора: заказ не создан). Шаг «сессия» при отсутствии `form_key` различает
Cloudflare-challenge/403 от прочего (по коду и телу) для понятной причины.

**Карточка** (`automations`): логин/пароль сайта (шифруются AES-256-GCM,
ключ — `AUTOMATION_KEY` или производный от `BOT_TOKEN`; наружу отдаётся только
маскированный логин и `has_creds`), количество бутылей, режим тары
(`auto` — из CRM / `fixed`), слот времени (`first`/конкретный), метод оплаты,
комментарий, интервал повтора (дней) и `next_run_at` (NULL — только вручную).

**Планировщик** (`internal/automation`, тик 1 мин): за сутки и за час до
`next_run_at` шлёт уведомление в бот (`notified_day`/`notified_hour`, чтобы не
дублировать), по наступлению срока — запускает; при успехе реального запуска
планирует следующий (`+interval_days`) и сбрасывает флаги. Ручной запуск и
dry-run идут через тот же `Execute` (лок `running` от параллельного запуска),
выполняются синхронно в HTTP-запросе, результат — в историю (`automation_runs`)
и уведомлением в бот. Первый реальный заказ рекомендуется сделать кнопкой
«Запустить» вручную.

## 2. Авторизация и роли

- Каждый запрос к `/api/v1/*` несёт `Authorization: tma <initData>`;
  подпись initData проверяется HMAC-SHA256 по алгоритму Telegram
  (`internal/auth/telegram.go`), initData старше 24 ч отклоняется.
- Раз в 5 минут на пользователя обновляется `last_seen_at` и фиксируется
  уникальная связка IP (X-Forwarded-For от Caddy) + устройство
  (короткая форма из User-Agent: «iPhone · Telegram»).
- **Роли**: ID админов — в конфиге сервера (`ADMIN_IDS`, `/opt/habits/.env`).
  Админ видит в Settings список пользователей (последний заход, устройство,
  IP), карточку пользователя (первый/последний заход, все связки IP+устройство
  с датой регистрации, счётчики данных по страницам — без самих данных)
  и может банить/разбанивать. Забаненный получает 403 на весь API
  (действует немедленно). Локальная разработка: `DEV_AUTH_BYPASS=true` +
  `Authorization: tma dev`.
